# 배치 처리 방식 트레이드오프 비교

## 결정 매트릭스

| 기준 | FastAPI BackgroundTasks | APScheduler |
|------|----------------------|-------------|
| **설정 복잡도** | 낮음 (내장) | 중간 (별도 설정) |
| **반복 실행** | 불가 (1회성) | 지원 (interval/cron) |
| **작업 지속성** | 없음 (메모리) | 선택적 (jobstore) |
| **동시성 제어** | 없음 | max_instances 지원 |
| **작업 취소** | 어려움 | 쉬움 (remove_job) |
| **모니터링** | 수동 구현 필요 | get_jobs() 제공 |
| **스케일 아웃** | 인스턴스별 독립 | 인스턴스별 독립 |
| **장애 복구** | 없음 | jobstore 설정 시 가능 |

---

## FastAPI BackgroundTasks

### 장점
- **제로 설정**: FastAPI에 내장되어 있어 별도 의존성 없음
- **간단한 사용법**: `background_tasks.add_task(func, *args)`로 즉시 사용
- **요청-응답 연동**: HTTP 요청 컨텍스트에서 자연스럽게 백그라운드 작업 추가
- **Starlette 기반**: 안정적인 내부 구현 (asyncio.ensure_future)

### 단점
- **1회성 실행만 가능**: 반복(interval), 예약(cron) 실행 불가
- **작업 관리 없음**: 취소, 재시도, 상태 조회 기능 미제공
- **장애 복구 불가**: 서버 재시작 시 진행 중인 작업 소실
- **동시성 제어 없음**: 동일 작업의 중복 실행을 막을 수 없음

### 적합한 케이스
- 이메일 발송, 알림 전송 등 1회성 비동기 작업
- 요청 처리 후 정리 작업 (로그 기록, 캐시 무효화)
- 응답 지연을 줄이기 위한 무거운 후처리

### 부적합한 케이스
- 매시간/매일 실행해야 하는 정기 작업
- 실패 시 자동 재시도가 필요한 작업
- 서버 재시작 후에도 이어서 처리해야 하는 작업

## APScheduler

### 장점
- **다양한 트리거**: interval, cron, date 트리거 지원
- **동시성 제어**: `max_instances`로 중복 실행 방지
- **작업 관리**: add_job, remove_job, get_jobs, pause/resume 등
- **AsyncIOScheduler**: FastAPI의 asyncio 이벤트 루프와 자연스럽게 통합
- **jobstore 플러그인**: SQLAlchemy, MongoDB, Redis 등 영속화 가능

### 단점
- **별도 의존성**: apscheduler 패키지 추가 필요
- **단일 인스턴스 제한**: 기본적으로 인스턴스별 독립 실행 (분산 잠금 필요)
- **설정 복잡도**: 트리거, 잡스토어, 실행기 설정이 필요
- **메모리 사용**: 장기 실행 시 작업 이력이 메모리에 누적

### 적합한 케이스
- 정기적인 데이터 집계, 리포트 생성
- 주기적 외부 API 폴링, 캐시 갱신
- 예약된 시간에 실행해야 하는 작업 (새벽 정산 등)
- 일정 간격 배치 처리 (로그 정리, 임시 파일 삭제)

### 부적합한 케이스
- 수천 개 이상의 분산 작업 처리 (Celery 권장)
- 멀티 인스턴스 환경에서 정확히 1번만 실행 보장 (분산 잠금 없이)
- 복잡한 워크플로우 오케스트레이션 (Airflow 권장)

## 언제 어떤 방식을 써야 하는가

```
요청에 의해 트리거되는 1회성 작업?
  → BackgroundTasks

주기적/예약 실행이 필요한 작업?
  → APScheduler

분산 환경 + 높은 신뢰성이 필요?
  → Celery + Redis/RabbitMQ (이 프로젝트 범위 외)

복잡한 DAG 워크플로우?
  → Airflow / Prefect (이 프로젝트 범위 외)
```

## 프로덕션 주의사항

### 1. 서버 재시작 시 작업 소실
BackgroundTasks와 APScheduler(메모리 jobstore) 모두 서버 재시작 시 진행 중인 작업이 소실됩니다.
- **대응**: 중요 작업은 DB에 상태를 기록하고, 서버 시작 시 미완료 작업을 재개하는 로직 구현
- **APScheduler**: `SQLAlchemyJobStore`를 사용하면 등록된 스케줄 자체는 영속화 가능

### 2. 멀티 인스턴스 환경에서 중복 실행
Gunicorn + uvicorn worker 구조에서 APScheduler가 각 워커마다 독립 실행되면 동일 작업이 N번 실행됩니다.
- **대응**: 스케줄러를 별도 프로세스로 분리하거나, Redis 기반 분산 잠금 사용
- **간단한 방법**: `--workers 1`로 단일 워커 실행 (소규모 서비스)

### 3. 메모리 누수 방지
인메모리 저장소 사용 시 작업 이력이 무한히 쌓이면 OOM 발생 가능합니다.
- **대응**: `MAX_HISTORY` 제한 (이 프로젝트: 50건), 완료 작업 주기적 정리

### 4. 에러 전파 주의
BackgroundTasks에서 발생한 예외는 클라이언트에게 전파되지 않습니다.
- **대응**: 작업 상태(status=failed, error 메시지)를 저장소에 기록하고, 별도 조회 API로 확인

### 5. 긴 실행 작업의 타임아웃
배치 작업이 예상보다 오래 걸릴 경우 리소스를 점유합니다.
- **대응**: 작업별 타임아웃 설정, 청크 단위 처리로 중간 상태 저장

## 이 프로젝트의 구현 결정 근거

| 결정 | 근거 |
|------|------|
| 인메모리 저장소 사용 | 학습 목적 프로젝트로 DB 의존성 최소화. MAX_HISTORY=50으로 메모리 제한 |
| BackgroundTasks + APScheduler 병행 | 두 방식의 차이를 직접 비교 학습할 수 있도록 구현 |
| 청크 단위 처리 | 진행률 추적 가능, 부분 실패 시 어디까지 처리되었는지 확인 가능 |
| fail_rate 파라미터 | 실패 시나리오를 인위적으로 재현하여 에러 처리 로직 검증 |
| max_instances=1 | 스케줄러 배치의 중복 실행 방지 (이전 실행이 끝나지 않으면 다음 실행 건너뜀) |
| scheduler_job_id 프리픽스 | `batch_` 접두사로 다른 스케줄러 작업과 네임스페이스 분리 |
